1. المقدمة
تُعد الأنظمة الموزعة أحد أهم مجالات هندسة البرمجيات الحديثة، حيث تعتمد على تشغيل عدة عقد (Nodes) تعمل معًا لتقديم خدمة موحدة للمستخدم النهائي.
في هذا المشروع تم بناء نظام مستشفى موزع لمحاكاة عملية حجز المواعيد الطبية باستخدام عدة خوادم مستقلة متصلة عبر تقنية Java RMI.
يركز المشروع على ضمان استمرار الخدمة حتى عند تعطل بعض الخوادم مع المحافظة على الأداء وتوزيع الحمل.
________________________________________
2. هدف المشروع
تم تنفيذ المشروع لتحقيق الأهداف التالية:
• إنشاء بيئة موزعة متعددة العقد.
• توزيع الطلبات على الخوادم.
• تقليل تأثير الأعطال.
• تحقيق استمرارية الخدمة.
• اختبار الاتساق النهائي.
• قياس الأداء.
________________________________________
3. وصف المشكلة
في الأنظمة التقليدية:
Client
↓
Single Server
↓
Failure
↓
Service Down
أما في النظام الحالي:
Client
↓
Load Balancer
↙ ↓ ↘
SERVER1 SERVER2 SERVER3
↓
Replication
↓
Continue Service
📷  
الشكل (1): البنية العامة للنظام الموزع.
________________________________________
4. تصميم النظام (System Architecture)
يتكون النظام من:
1.	Client
2.	Load Balancer
3.	Distributed Servers
4.	Health Monitor
5.	Replication Layer
طريقة التنفيذ:
•	العميل يرسل الطلب.
•	الموازن يحدد الخادم.
•	يتم تنفيذ الحجز.
 
الشكل (2): الهيكل البرمجي للمشروع.
________________________________________
5. التنفيذ العملي
5.1 الاتصال الموزع (RMI)
تم إنشاء واجهة مشتركة:
AppointmentService.java
السماح للعميل باستدعاء وظائف موجودة على خادم آخر.
📷  
الشكل (3): تشغيل العقد الموزعة.
________________________________________
5.2 توزيع الأحمال (Load Balancing)
تم استخدام:
Round Robin
مثال:
PATIENT-1 → SERVER-1
PATIENT-2 → SERVER-3
PATIENT-3 → SERVER-1
الهدف:
•	توزيع الحمل.
•	تحسين الاستجابة.
📷  
الشكل (4): توزيع الطلبات بين الخوادم.
________________________________________
5.3 تحمل الأعطال (Fault Tolerance)
تم إيقاف أحد الخوادم عمدًا.
النتيجة:
SERVER-2 OFFLINE
لكن النظام استمر.
.
________________________________________
5.4 فحص الحالة (Health Check)
قبل تنفيذ الطلب:
health()
↓
UP
↓
Execute
إذا:
DOWN
↓
Skip
📷  
الشكل (5): فحص حالة الخادم.
________________________________________
5.5 النسخ (Leader–Follower)
تم استخدام قائد فعّال.
أمثلة:
[LEADER=SERVER-1]
[LEADER=SERVER-3]
